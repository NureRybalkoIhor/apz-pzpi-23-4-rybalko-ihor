3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Шаблони проєктування - це високорівневі, перевірені часом архітектурні рішення для типових проблем у розробці ПЗ. Вони не є готовими фрагментами коду, а слугують концептуальними кресленнями. Згідно з класифікацією каталогу GoF («Банди чотирьох»), патерни поділяються на три основні групи: породжувальні (відповідають за гнучке створення об'єктів), структурні (формують зв'язки та ієрархії між класами) та поведінкові (визначають способи комунікації між об'єктами). Основною областю застосування патернів є уніфікація кодової бази, зниження зв'язності (loose coupling) та створення спільного словника для комунікації всередині команди розробників.
3.2 Опис обраного патерна
Назва патерна: Facade (Фасад)
Тип шаблона (патерна): структурний (Structural pattern)
Основне призначення: надання простого, уніфікованого інтерфейсу до складної системи об'єктів, фреймворку або сторонньої бібліотеки з метою приховування внутрішньої складності від клієнтського коду.
Передумови застосування: використання є доцільним за наявності складної підсистеми з великою кількістю взаємопов'язаних класів; при необхідності зменшити кількість прямих залежностей між бізнес-логікою та інфраструктурними сервісами; а також для розшарування архітектури (Facade як точка входу на рівень).
Переваги: надійна інкапсуляція складності підсистеми; кардинальне зниження зв'язності (клієнт залежить лише від одного класу); спрощення модульного тестування (Unit-тестів) клієнтського коду завдяки можливості використання єдиного Mock-об'єкта.
Недоліки: ризик перетворення Фасаду на антипатерн «Божественний об'єкт» (God Object), якщо він почне акумулювати в собі надмірну кількість бізнес-логіки замість простого делегування.
3.3 Структура шаблона (патерна)
Опис структурних частин:
* Клієнт (Client) - ініціатор запиту, який взаємодіє виключно з Фасадом, не маючи доступу до внутрішніх класів.
* Фасад (Facade) - проміжний клас-координатор. Знає, яким саме класам підсистеми делегувати виконання задачі та керує процесом їхньої взаємодії.
* Складна підсистема (Subsystems) - набір незалежних класів (бази даних, сервіси, алгоритми), які виконують фактичну роботу. Підсистеми не знають про існування Фасаду.
* Додатковий Фасад (Additional Facade) - опціональний компонент для розділення обов'язків, якщо основний Фасад стає занадто великим.
Опис взаємодії об’єктів: клієнт звертається до Фасаду через простий метод. Фасад розпізнає запит, ініціалізує необхідні об'єкти підсистеми, передає їм дані та визначає порядок виконання операцій. Після виконання транзакції всередині підсистеми, результати (за необхідності) повертаються Клієнту через Фасад.
UML-діаграми з поясненнями: класична UML-діаграма патерна (див. рис. 1) демонструє спрямовану асоціацію (суцільну стрілку) від Клієнта виключно до Фасаду. Фасад має асоціативні зв'язки (пунктирні стрілки) з об'єктами підсистеми. Важливим поясненням є відсутність зворотних зв'язків: від класів підсистеми не йдуть лінії ні до Фасаду, ні до Клієнта, що підтверджує їхню незалежність.

3.4 Реалізація шаблона (патерна)
Опис реалізації мовою програмування: C# (.NET)
Основні етапи:
1. Створення базових інтерфейсів/класів: розробка незалежних компонентів системи гейміфікованого трекінгу звичок: DatabaseContext (збереження прогресу), GamificationEngine (розрахунок досвіду), NotificationService (відправка Push-сповіщень).
2. Реалізація конкретних класів: створення координаційного класу HabitTrackerFacade, який інкапсулює залежності від перелічених вище сервісів та надає єдиний метод CompleteDailyHabit().
3. Демонстрація використання: написання коду HabitController (роль Клієнта), який замість багаторядкової транзакції викликає збереження прогресу, нарахування балів та сповіщення за допомогою одного делегованого виклику до Фасаду.

4 ВИСНОВКИ
У результаті виконання практичної роботи було досліджено структурний шаблон проєктування Facade (Фасад). Проаналізовано його ключові архітектурні особливості, зокрема механізм надання уніфікованого інтерфейсу до складної підсистеми. Визначено, що застосування цього патерна забезпечує зниження зв'язності (loose coupling) компонентів та ефективно ізолює клієнтський код від деталей реалізації внутрішніх сервісів.
Практична частина роботи полягала у програмній реалізації патерна мовою C# (.NET) на прикладі системи гейміфікованого трекінгу звичок. Розроблено код, який демонструє інкапсуляцію взаємодії кількох модулів (база даних, ігровий рушій, сервіс сповіщень) через єдиний клас-координатор. Виконання цього завдання дозволило сформувати навички застосування патернів проєктування для спрощення архітектури та полегшення подальшого модульного тестування системи.
У межах виконання організаційних вимог створено GitHub-репозиторій із стандартизованою ієрархічною структурою та базовою документацією. Додатково підготовлено звітні матеріали: мультимедійну презентацію та відеодоповідь, що закріплює навички технічної комунікації та презентації архітектурних рішень.
Загалом, результати роботи підтверджують, що практичне застосування шаблонів проєктування є необхідним стандартом у розробці програмного забезпечення. Впровадження патернів дозволяє суттєво підвищити рівень повторного використання коду, покращує загальну масштабованість та полегшує довгострокову підтримку програмних систем.

В.2 Довідка щодо генерації кодової реалізації класу-Фасаду за допомогою ШІ
Назва інструменту та версія: Gemini 3.1 Pro.
Мета використання: для створення структурного класу-координатора (Facade), який інкапсулює логіку взаємодії створених раніше підсистем за єдиним спрощеним інтерфейсом.
Внесок автора: згенерований код було адаптовано для наочної демонстрації агрегації залежностей (ініціалізація підсистем всередині класу). Логіку єдиного методу було відредаговано для відображення строгих кроків бізнес-транзакції (БД -> Бали -> Сповіщення) без перевантаження синтаксичними деталями.
Промпт: «Дій як Software Architect. На основі створених підсистем (DatabaseContext, GamificationEngine, NotificationService) напиши клас HabitTrackerFacade мовою C#. Цей клас повинен мати один публічний метод CompleteDailyHabit, який ховає в собі послідовний виклик методів підсистем. Продемонструй принцип диригента.»
Результат (приклад програмного коду):
  1 public class HabitTrackerFacade
  2 {
  3     private DatabaseContext _db = new DatabaseContext();
  4     private GamificationEngine _game = new GamificationEngine();
  5     private NotificationService _notify = new NotificationService();
  6
  7     public void CompleteDailyHabit(int userId, int habitId)
  8     {
  9         _db.SaveProgress(habitId);
 10
 11         int xp = _game.CalculateXP();
 12         _game.AddXP(userId, xp);
 13
 14         _notify.SendPush(userId, $"Звичку виконано! +{xp} XP");
 15     }
 16 }
