2 ЗАВДАННЯ
Завданням поточної лабораторної роботи є розробка мобільного програмного застосунку для однієї з поширених мобільних платформ. Застосунок повинен бути частиною більшої системи та взаємодіяти з її іншими компонентами. Для реалізації можна використовувати Kotlin для Android або Swift для iOS.
Звіт повинен містити опис прийнятих інженерних рішень, будову застосунку та опис його взаємодії з іншими частинами системи. Обов'язковими є UML діаграма прецедентів та UML діаграма компонент. Окрім цього, необхідно додати не менше двох діаграм на вибір із наступних: ER-модель даних мобільної платформи, UML діаграма пакетів, UML діаграма взаємодії, UML діаграма діяльності або UML діаграма станів. Код застосунку необхідно завантажити на GitHub до захисту роботи.

3 ОПИС ВИКОНАНОЇ РОБОТИ
Під час виконання лабораторної роботи було вирішено завдання проєктування та реалізації мобільного програмного застосунку для платформи попереднього замовлення їжі FoodPreOrder. На цьому етапі розробка зосередилася на побудові клієнтського інтерфейсу для кінцевого користувача (гостя закладу), реалізації локальної бізнес-валідації та забезпеченні безперервної взаємодії мобільного клієнта із серверною частиною системи через RESTful API. Розробка велася на мові Kotlin із використанням декларативного UI-фреймворку Jetpack Compose (Material 3) та архітектурного патерну MVVM (Model-View-ViewModel), що дозволило забезпечити чітке розмежування відповідальності між шарами презентації, бізнес-логіки та даних.
Реалізація мобільного застосунку охопила сценарії взаємодії клієнта із системою: від реєстрації та авторизації до оформлення попереднього замовлення, онлайн-оплати та відстеження статусу приготування в реальному часі. Також було розроблено підсистему локальних сповіщень, яка забезпечує інформування користувача про зміну статусу замовлення, реалізовану через механізм фонового опитування серверу та системні Android-нотифікації.
3.1 Архітектура мобільного застосунку
Архітектура застосунку побудована за багатошаровим принципом із розмежуванням відповідальності, що структурно проєкт складається з п'яти логічних шарів.
Шар презентації (Presentation Layer) реалізований за допомогою Jetpack Compose та містить екрани користувацького інтерфейсу: SplashScreen, LoginScreen, RegisterScreen, ForgotPasswordScreen, HomeScreen, MenuScreen, CartScreen, PaymentScreen, OrderTrackingScreen, OrderHistoryScreen, NotificationsScreen та ProfileScreen. Навігація між екранами здійснюється через компонент NavHost із параметризованими маршрутами, визначеними у sealed-класі Screen.
Шар ViewModel відповідає за управління станом UI та інкапсуляцію бізнес-логіки презентації. Реалізовано шість ViewModel-класів: AuthViewModel (автентифікація), HomeViewModel (список ресторанів), MenuViewModel (меню та кошик), OrderViewModel (життєвий цикл замовлення), NotificationViewModel (сповіщення) та ProfileViewModel (профіль). Стан кожного ViewModel моделюється через sealed-класи (наприклад, AuthState, OrderState), що забезпечує типобезпечну обробку всіх можливих станів екрану: завантаження, успіх, помилка. Реактивність досягається через StateFlow - потоки даних, на які підписуються Composable-функції.
Шар репозиторіїв (Repository Layer) виступає єдиною точкою доступу до даних для ViewModel та абстрагує джерела інформації. Реалізовано п'ять репозиторіїв: AuthRepository, RestaurantRepository, DishRepository, OrderRepository та NotificationRepository. Кожен репозиторій інкапсулює логіку формування HTTP-заголовків авторизації (додавання префіксу «Bearer» до JWT-токена).
Мережевий шар (Network Layer) побудований на бібліотеці Retrofit із серіалізатором Gson. Централізований об'єкт RetrofitClient ініціалізує HTTP-клієнт та створює реалізації п'яти інтерфейсів API-сервісів: AuthApiService, RestaurantApiService, DishApiService, OrderApiService та NotificationApiService. Базова URL-адреса налаштована на спеціальний loopback-адрес емулятора Android (10.0.2.2) для взаємодії з локальним сервером розробки.
Шар локальних даних (Local Data Layer) представлений класом SessionManager, який використовує Jetpack DataStore Preferences для персистентного зберігання сесії користувача (JWT-токен, роль, ім'я, email) та опціональних платіжних даних. Усі операції читання реалізовані через реактивні Flow-потоки, що дозволяє UI автоматично реагувати на зміни стану авторизації.
3.2 Реалізація бізнес-логіки мобільного клієнта
Ключовою архітектурною особливістю застосунку є централізований модуль валідації (клас Validators), який інкапсулює всі бізнес-правила на стороні клієнта в чотирьох спеціалізованих об'єктах-валідаторах. Кожен валідатор повертає типобезпечний результат через sealed-клас ValidationResult (Ok або Fail із текстом помилки), що уніфікує обробку помилок на всіх екранах.
3.2.1 Валідація автентифікації (AuthValidator)
Об'єкт AuthValidator забезпечує перевірку всіх полів форм реєстрації та входу. Валідація email-адреси виконується за допомогою регулярного виразу, що перевіряє наявність символу «@», доменної частини та зони верхнього рівня довжиною щонайменше 2 символи. Номер телефону перевіряється на відповідність міжнародному формату (від 10 до 15 цифр із опціональним префіксом «+»). Пароль повинен містити щонайменше 6 символів. Повне ім'я перевіряється на відсутність цифрових символів та мінімальну довжину 2 знаки.
Метод validateRegistration реалізує композитну перевірку: він послідовно застосовує п'ять валідаторів (ім'я, email, телефон, пароль, підтвердження пароля) та повертає першу знайдену помилку, забезпечуючи принцип «fail-fast» - користувач бачить лише найпершу проблему, що потребує виправлення.


3.2.2 Валідація кошика та замовлення (CartValidator)
Об'єкт CartValidator реалізує набір бізнес-правил, що регулюють процес формування замовлення. Визначено чотири порогові константи: максимальна кількість одиниць однієї страви (MAX_QUANTITY_PER_DISH = 20), максимальна загальна кількість позицій у замовленні (MAX_CART_ITEMS = 30), мінімальна сума замовлення (MIN_ORDER_AMOUNT = 30 грн) та обов'язкова прив'язка до конкретного ресторану (restaurantId > 0).
Метод validateBeforeOrder виконує комплексну перевірку перед відправкою замовлення на сервер, послідовно застосовуючи чотири правила: непорожність кошика, обмеження розміру, мінімальна сума та консистентність ресторану.
3.2.3 Валідація часу візиту (OrderTimeValidator)
Цей валідатор реалізує найскладнішу бізнес-логіку мобільного клієнта - динамічний розрахунок допустимого вікна часу візиту. Мінімальний час візиту обчислюється на основі найдовшого часу приготування серед усіх страв у кошику, але не менше 15 хвилин. Формально мінімальний час T_min визначається як поточний час плюс максимум із двох значень: найбільшого часу приготування страви у кошику та буферного інтервалу 15 хвилин. Максимальний горизонт замовлення обмежений трьома днями вперед.
Метод validate виконує трирівневу перевірку обраного часу: заборона вибору минулого часу, порівняння з динамічно обчисленим мінімумом (із вказівкою конкретної страви, що потребує найбільше часу) та порівняння з максимальним горизонтом.

3.2.4 Валідація платіжних даних (PaymentValidator)
Об'єкт PaymentValidator забезпечує клієнтську верифікацію банківських реквізитів. Номер картки перевіряється на довжину (16 цифр), наявність лише цифрових символів та коректність за алгоритмом Луна (Luhn algorithm). Алгоритм Луна реалізовано як ітеративну функцію luhnCheck, яка обходить цифри номера справа наліво, подвоюючи кожну другу цифру та обчислюючи контрольну суму. Картка вважається валідною, якщо залишок від ділення контрольної суми на 10 дорівнює нулю.
Термін дії картки перевіряється за регулярним виразом (формат ММ/РР) та порівнюється з поточною датою для виявлення прострочених карток. CVV-код перевіряється на довжину (3-4 цифри) та наявність лише цифрових символів.
3.3 Механізм відстеження замовлення в реальному часі
Для забезпечення безперервного моніторингу стану замовлення реалізовано механізм короткого опитування (short polling). Метод trackOrder у класі OrderViewModel запускає нескінченний цикл у корутині viewModelScope, який кожні 10 секунд надсилає GET-запит до серверного ендпоінту /api/Orders/{id}. Отриманий стан замовлення оновлює StateFlow, що автоматично перемальовує UI екрану трекінгу.
Цикл опитування автоматично завершується при досягненні фінальних статусів (Completed або Cancelled), що запобігає непотрібному навантаженню на мережу та батарею пристрою. Для обробки мережевих помилок використовується ігнорування виключень із продовженням циклу, що забезпечує стійкість до короткочасних збоїв з'єднання.

3.4 Підсистема push-сповіщень
У класі MainActivity реалізовано фоновий сервіс сповіщень, який працює паралельно з основним інтерфейсом. Механізм запускається в lifecycleScope активності та кожні 15 секунд опитує серверний ендпоінт /api/Notifications на наявність непрочитаних повідомлень. При виявленні нових сповіщень система генерує Android-нотифікації через клас NotificationService, який створює виділений канал сповіщень (order_status_channel) із високим пріоритетом (IMPORTANCE_HIGH) та автоматично позначає оброблені повідомлення як прочитані на сервері.
Додатково реалізовано внутрішній екран сповіщень (NotificationsScreen), де користувач може переглянути повну історію повідомлень, позначити окреме сповіщення як прочитане або виконати масову операцію «Прочитати все». Список сповіщень автоматично сортується за датою відправки у зворотному порядку.
3.5 UML-діаграми мобільного застосунку
3.5.1 Діаграма прецедентів (Use Case Diagram)
Побудована діаграма прецедентів (див. додаток Б.1) відображає повний перелік функціональних можливостей мобільного застосунку з точки зору кінцевого користувача. На діаграмі визначено двох акторів: Клієнт (основний актор, який ініціює всі бізнес-сценарії) та REST API Сервер (зовнішня система, з якою взаємодіє кожен прецедент). Системна межа окреслює 16 прецедентів, що охоплюють увесь функціонал застосунку: реєстрація, авторизація, відновлення паролю, перегляд ресторанів, перегляд меню ресторану, фільтрація за категоріями, додавання страви до кошика, управління кошиком, вибір часу візиту, оформлення замовлення, оплата замовлення, відстеження статусу замовлення, скасування замовлення, перегляд історії замовлень, перегляд сповіщень та перегляд профілю. Між прецедентами визначено зв'язки типу include: оформлення замовлення обов'язково включає вибір часу візиту, а оплата замовлення включає подальше відстеження статусу.
3.5.2 Діаграма компонентів (Component Diagram)
Діаграма компонентів (див. додаток Б.2) візуалізує внутрішню архітектуру мобільного застосунку та його зовнішні залежності. Основний пакет Mobile App містить компоненти, організовані за п'ятьма шарами. Шар Presentation включає компоненти UI Screens (Jetpack Compose) та Navigation (AppNavGraph). Шар ViewModel містить шість компонентів: AuthViewModel, HomeViewModel, MenuViewModel, OrderViewModel, NotificationVM та ProfileVM. Шар Repository представлений п'ятьма компонентами: AuthRepository, RestaurantRepository, DishRepo, OrderRepository та NotificationRepo. Мережевий шар включає RetrofitClient (HTTP Client із серіалізатором Gson) та API Services (Retrofit Interfaces). Шар локальних даних містить SessionManager (DataStore Preferences) та Validators (Business Rules). Окремо виділено компонент NotificationService (Android NotificationChannel).
Зовнішній пакет Server відображає серверну інфраструктуру, з якою взаємодіє застосунок: ASP.NET Core (RESTful API), MSSQL Database (Entity Framework Core) та Azure Blob Storage (Media Content). Зв'язок між мобільним клієнтом та сервером здійснюється через протокол HTTP/JSON із авторизацією Bearer JWT.
3.5.3 Діаграма діяльності (Activity Diagram)
Діаграма діяльності (див. додаток Б.3) моделює основний бізнес-процес застосунку - повний цикл попереднього замовлення їжі від запуску застосунку до отримання замовлення у ресторані. Процес починається з екрану SplashScreen, де відбувається перше розгалуження: система перевіряє наявність збереженого JWT-токена у DataStore. При відсутності токена користувач перенаправляється на екран авторизації або реєстрації, при наявності - одразу на головний екран.
Далі процес проходить через послідовні етапи: перегляд списку ресторанів, вибір конкретного закладу, перегляд його меню з можливістю фільтрації за категоріями та додавання страв до кошика. На етапі кошика відбувається друге розгалуження - валідація бізнес-правил (мінімальна сума, ліміти кількості). У разі помилки валідації користувач отримує повідомлення та повертається до редагування кошика.
Після успішної валідації користувач обирає час візиту (з урахуванням часу приготування) та опціонально додає коментар. Оформлення замовлення відправляє POST-запит до сервера. На етапі трекінгу відбувається третє розгалуження: користувач може або оплатити замовлення (ввівши дані картки), або скасувати його (з підтвердженням незворотності дії). Після оплати процес переходить до відстеження статусу з полінгом кожні 10 секунд та завершується отриманням замовлення у ресторані.
3.5.4 Діаграма станів (State Diagram)
Діаграма станів (див. додаток Б.4) моделює життєвий цикл сутності замовлення (Order) з точки зору мобільного клієнта. Початковий стан визначається подією createOrder(), яка переводить замовлення у стан Pending. У цьому стані система очікує оплати від клієнта, при цьому доступні два переходи: payOrder() (при успішній оплаті) переводить замовлення у стан Paid, а cancelOrder() - у фінальний стан Cancelled.
Зі стану Paid замовлення переходить у стан Cooking після прийняття його кухнею. У цьому стані визначено рефлексивний перехід trackOrder(), який виконується кожні 10 секунд для оновлення інформації на екрані клієнта. Коли страви готові, замовлення переходить у стан Ready, при якому система автоматично генерує push-сповіщення клієнту з повідомленням про готовність. Після фактичного отримання замовлення клієнтом стан змінюється на фінальний - Completed.

3.6 Використання засобів ШІ

3.6.1 Довідка щодо генерації модуля клієнтської валідації платіжних даних за допомогою ШІ 
Назва інструменту та версія: Claude Sonnet 4.6 (Thinking).
Мета використання: для реалізації алгоритму Луна (Luhn algorithm) з метою верифікації номерів банківських карток на стороні мобільного клієнта та побудови композитного валідатора платіжних реквізитів.

Під час виконання лабораторної роботи було проведено повний цикл проєктування та програмної реалізації мобільного застосунку для платформи попереднього замовлення їжі FoodPreOrder. Основною метою роботи було створення зручного та надійного клієнтського інтерфейсу, який забезпечує безперебійну взаємодію кінцевого користувача із серверною частиною системи та реалізує ключові бізнес-сценарії: від перегляду каталогу ресторанів до оплати та відстеження замовлення в реальному часі. У процесі розробки було застосовано сучасні підходи до побудови Android-застосунків, включаючи архітектурний патерн MVVM, декларативний UI-фреймворк Jetpack Compose та реактивне управління станом через Kotlin StateFlow.
Ключовим етапом роботи стала реалізація централізованого модуля бізнес-валідації, який інкапсулює критичні правила системи безпосередньо на стороні клієнта. Такий підхід забезпечує миттєвий зворотний зв'язок для користувача та зменшує навантаження на сервер за рахунок фільтрації некоректних запитів ще до їх відправки.
У рамках забезпечення інформування користувача було реалізовано дворівневу систему сповіщень: фоновий сервіс у MainActivity, який кожні 15 секунд опитує API та генерує системні Android-нотифікації через виділений канал високого пріоритету, та внутрішній екран із повною історією повідомлень та можливістю масового позначення як прочитаних.   1 object PaymentValidator {
  2
  3     private val EXPIRY_REGEX = Regex("^(0[1-9]|1[0-2])/([0-9]{2})$")
  4
  5     fun validateCardNumber(number: String): ValidationResult = when {
  6         number.isBlank()     -> ValidationResult.Fail("Введіть номер картки")
  7         number.length != 16  -> ValidationResult.Fail("Номер картки має містити 16 цифр")
  8         !number.all { it.isDigit() } -> ValidationResult.Fail("Номер картки має містити лише цифри")
  9         !luhnCheck(number)   -> ValidationResult.Fail("Невірний номер картки (перевірка Luhn)")
 10         else                 -> ValidationResult.Ok
 11     }
 12
 13     fun validateExpiry(expiry: String): ValidationResult {
 14         if (!EXPIRY_REGEX.matches(expiry)) return ValidationResult.Fail("Формат: ММ/РР")
 15         val (month, year) = expiry.split("/")
 16         val now = java.time.LocalDate.now()
 17         val cardYear = 2000 + year.toInt()
 18         val cardMonth = month.toInt()
 19         return when {
 20             cardYear < now.year -> ValidationResult.Fail("Картка прострочена")
 21             cardYear == now.year && cardMonth < now.monthValue -> ValidationResult.Fail("Картка прострочена")
 22             else -> ValidationResult.Ok
 23         }
 24     }
 25
 26     fun validateCvv(cvv: String): ValidationResult = when {
 27         cvv.isBlank()        -> ValidationResult.Fail("Введіть CVV")
 28         cvv.length !in 3..4  -> ValidationResult.Fail("CVV має містити 3-4 цифри")
 29         !cvv.all { it.isDigit() } -> ValidationResult.Fail("CVV має містити лише цифри")
 30         else                 -> ValidationResult.Ok
 31     }
 32
 33     /** Алгоритм Луна для перевірки номера картки */
 34     fun luhnCheck(number: String): Boolean {
 35         var sum = 0
 36         var alternate = false
 37         for (i in number.length - 1 downTo 0) {
 38             var n = number[i].digitToInt()
 39             if (alternate) {
 40                 n *= 2
 41                 if (n > 9) n -= 9
 42             }
 43             sum += n
 44             alternate = !alternate
 45         }
 46         return sum % 10 == 0
 47     }
 48
 49     fun validateAll(cardNumber: String, expiry: String, cvv: String): ValidationResult =
 50         listOf(
 51             validateCardNumber(cardNumber),
 52             validateExpiry(expiry),
 53             validateCvv(cvv)
 54         ).firstOrNull { it is ValidationResult.Fail } ?: ValidationResult.Ok
 55 }

  2
  3     const val MAX_DAYS_AHEAD = 3L
  4     const val MIN_BUFFER_MINUTES = 15L
  5
  6     fun calcMinVisitTime(cart: List<CartItem>): LocalDateTime {
  7         val maxPrep = cart.maxOfOrNull { it.preparationTimeMinutes }?.toLong() ?: MIN_BUFFER_MINUTES
  8         return LocalDateTime.now().plusMinutes(maxOf(maxPrep, MIN_BUFFER_MINUTES))
  9     }
 10
 11     fun calcMaxVisitTime(): LocalDateTime = LocalDateTime.now().plusDays(MAX_DAYS_AHEAD)
 12
 13     fun validate(visitTime: LocalDateTime?, cart: List<CartItem>): ValidationResult {
 14         if (visitTime == null) return ValidationResult.Fail("Оберіть час візиту")
 15
 16         val minTime = calcMinVisitTime(cart)
 17         val maxTime = calcMaxVisitTime()
 18
 19         if (visitTime.isBefore(LocalDateTime.now()))
 20             return ValidationResult.Fail("Не можна замовляти в минулому")
 21
 22         if (visitTime.isBefore(minTime)) {
 23             val maxPrep = cart.maxOfOrNull { it.preparationTimeMinutes } ?: MIN_BUFFER_MINUTES.toInt()
 24             val slowest = cart.maxByOrNull { it.preparationTimeMinutes }?.nameUA ?: ""
 25             return ValidationResult.Fail(
 26                 "Занадто рано! «$slowest» готується $maxPrep хв — " +
 27                 "найраніший можливий час: ${minTime.toLocalTime().toString().take(5)}"
 28             )
 29         }
 30
 31         if (visitTime.isAfter(maxTime))
 32             return ValidationResult.Fail("Не можна замовляти більше ніж на $MAX_DAYS_AHEAD дні вперед")
 33
 34         return ValidationResult.Ok
 35     }
 36 }
