	2 ЗАВДАННЯ
Метою роботи є дослідження архітектури складної програмної системи та аналіз застосованих архітектурних підходів.
Необхідно підготувати доповідь про архітектуру однієї з відомих складних програмних систем (операційна система, платформа, вебсервіс, розподілена система, середовище розробки, база даних, хмарна платформа тощо). 
У межах роботи необхідно:
* виконати аналіз архітектури програмної системи;
* визначити архітектурний стиль;
* описати основні компоненти;
* проаналізувати взаємодію підсистем;
* оцінити масштабованість та переваги архітектурних рішень;
* підготувати презентацію та відеодоповідь;
* оформити звіт відповідно до ДСТУ 3008:2015.




3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: Penpot.
Розробник: Kaleidos Open Source (Мадрид, Іспанія).
Призначення системи: відкрита (Open Source) кросплатформна вебсистема для створення UI/UX дизайну та інтерактивного прототипування. Головна філософія платформи - це "Design as Code" (Дизайн як код), що передбачає використання відкритих вебстандартів для мінімізації розриву між етапами проєктування інтерфейсу та його верстки.
Користувачі: UX/UI-дизайнери, фронтенд-розробники, продуктові менеджери та крос-функціональні (Agile) команди, які потребують єдиного середовища для колаборації.
Основні функції: повноцінне векторне редагування макетів безпосередньо у браузері; підтримка спільної роботи багатьох користувачів у реальному часі (Multiplayer); нативна підтримка сучасних стандартів CSS, таких як Flex Layout та CSS Grid; створення інтерактивних прототипів з переходами; експорт готових макетів у різні формати (PNG, PDF, SVG).
3.2 Архітектурний стиль системи
Система побудована за гібридною архітектурою, що поєднує клієнт-серверну модель типу SPA (Single Page Application) з центральним монолітним ядром та виділеними подієво-орієнтованими мікросервісами для ресурсомістких задач.
Ключовим викликом для систем такого класу є забезпечення миттєвої безконфліктної синхронізації дій десятків користувачів (Multiplayer). Для цього розробники обрали екосистему Clojure/ClojureScript, що дозволило реалізувати парадигму "Shared Code", що являє собою використання єдиної кодової бази для клієнта і сервера. Це повністю усуває дублювання бізнес-логіки та розбіжності у моделях даних. Водночас виділення важких процесів (наприклад, генерації PDF-файлів) в окремий асинхронний мікросервіс дозволяє ефективно масштабувати систему та уникати блокування основного монолітного бекенду під час пікових навантажень.
3.3 Основні компоненти системи
Система складається з кількох логічних та фізичних підсистем:
1. Інтерфейс користувача (frontend): односторінковий додаток (SPA), розроблений на ClojureScript з використанням обгортки Reagent (над бібліотекою React). Використовує підхід "SVG as DOM", де кожен елемент дизайну декларативно рендериться як стандартний тег браузера.
2. Серверна частина (backend): серверний моноліт, написаний на Clojure, що працює в середовищі JVM (Java Virtual Machine). Відповідає за обробку API-запитів, управління сесіями та валідацію дій користувачів.
3. Рівень бізнес-логіки (common Code): спільний модуль (.cljc), який містить незмінні (immutable) моделі даних, математичні трансформації SVG та правила валідації. Цей код компілюється одночасно у JavaScript (для UI) та Java Bytecode (для Backend API).
4. Рівень зберігання даних: Розподілений на три спеціалізовані сховища:
   1. PostgreSQL: транзакційна реляційна база для надійного зберігання структури користувачів, проєктів та векторних макетів.
   2. Redis: In-Memory база даних, що виконує роль брокера повідомлень (для маршрутизації подій) та швидкого сховища активних сесій.
   3. S3-сховище (MinIO/AWS): об'єктне сховище для зберігання "важких" бінарних файлів (завантажених зображень, кастомних шрифтів).
5. Допоміжні сервіси (exporter app): ізольований мікросервіс на базі Node.js. Використовує бібліотеку Puppeteer та прихований браузер (Headless Chrome) для точного асинхронного рендерингу складних векторних макетів у растрові формати.
3.4 Взаємодія компонентів
Взаємодія в системі має чітко розділений дворівневий характер. Стандартні, некритичні до часу операції (авторизація, налаштування профілю, управління теками) виконуються через класичні HTTP/REST запити. Натомість робота в режимі редактора (Multiplayer) здійснюється через постійне двостороннє з'єднання WebSockets. Передача даних про зміни в макеті базується на патерні Command через мережевий RPC (Remote Procedure Call): клієнт формує оптимізований JSON-об'єкт (патч), який містить лише змінені атрибути фігури. Синхронізація між різними клієнтами реалізована через патерн Pub/Sub: коли сервер отримує патч, він валідує його, зберігає у PostgreSQL і паралельно публікує подію в брокер Redis. Redis, у свою чергу, миттєво розсилає це оновлення всім іншим клієнтам, підключеним до цього ж проєкту, що забезпечує відображення змін у реальному часі. 
На рисунку 3.1 відображено контейнерну архітектуру Penpot. Показано розподіл на клієнтську частину (яка використовує основний потік React та фонові Web Workers), центральний Backend на JVM та мікросервіс Exporter. Стрілками візуалізовано інформаційні потоки: встановлення WebSocket-з'єднання з сервером, HTTP-виклики до Exporter, а також внутрішню взаємодію бекенду з PostgreSQL (читання/запис) та Redis (механізми Publish/Subscribe).

3.5 Масштабованість та надійність 
Завдяки ізоляції компонентів система підтримує горизонтальне масштабування. Балансувальники (наприклад, Nginx або HAProxy) можуть динамічно розподіляти трафік між кількома інстансами Backend-сервера або створювати додаткові копії мікросервісу Exporter App під час інтенсивного експорту макетів.
На рівні інтерфейсу застосовується підхід Optimistic UI: клієнт відображає зміни миттєво, приховуючи затримки мережі, та має модуль Auto-reconnect для відновлення роботи після обриву зв'язку. Транзакційність PostgreSQL гарантує, що при збої сервера дані не будуть збережені частково (макет або зберігається повністю, або зміни відкидаються).
Використання Redis для кешування станів авторизації робить основні вузли бекенду stateless (без збереження стану), що дозволяє безшовно перемикати користувачів між серверами при збоях.
Передбачена підтримка механізмів реплікації Master-Slave для PostgreSQL, що забезпечує резервування критичних даних користувачів.
Платформа повністю адаптована до хмарної інфраструктури. Усі компоненти розгортаються через Docker та підтримують оркестрацію за допомогою Kubernetes, що забезпечує ідентичність середовищ розробки, тестування та експлуатації.
3.6 Приклади програмної реалізації 
В основі архітектури Penpot лежить функціональна парадигма програмування. Наведені нижче словесні представлення з посиланням на фрагменти коду у додатках (псевдокод на основі реальної архітектури, мови Clojure/ClojureScript) демонструють ключові принципи обробки даних у системі: роботу з незмінними структурами, подієву маршрутизацію та декларативний рендеринг. 
3.6.1 Потокобезпечне управління станом (Atoms) (див. дод. В.1)
Архітектурний принцип: Immutable State & Concurrency. Використання незмінних структур даних та потокобезпечних "атомів" дозволяє бекенду обробляти сотні конкурентних запитів від користувачів без ризику виникнення стану гонитви (Race Conditions) та без застосування ресурсомістких блокувань (Mutex). 
3.6.2 Подієва синхронізація через Pub/Sub (Redis) (див. дод. В.2)
Архітектурний принцип: Event-Driven Architecture. Реалізація слабозв'язаної архітектури за допомогою брокера повідомлень для асинхронної трансляції змін усім учасникам сесії. 
3.6.3 Реактивний UI на основі відкритих вебстандартів (ClojureScript) (див. дод. В.3)
Архітектурний принцип: Declarative UI & SVG as DOM. На відміну від систем, що використовують "глухий" Canvas, Penpot генерує інтерфейс як набір нативних SVG-тегів. Це дозволяє браузеру ефективно відмальовувати лише ті елементи, властивості яких змінилися.

4 ВИСНОВКИ
У результаті виконання практичного заняття було досліджено архітектуру програмної системи Penpot - сучасної відкритої платформи для UI/UX дизайну та прототипування. Під час роботи було визначено, що система базується на гібридному архітектурному стилі. Цей підхід ефективно поєднує стабільність монолітного серверного ядра, побудованого на екосистемі JVM (мова Clojure), із гнучкістю ізольованого мікросервісу (Exporter App на Node.js), який бере на себе виконання найбільш ресурсомістких задач з рендерингу графіки.
Окрему увагу було приділено аналізу взаємодії компонентів системи. Встановлено, що для забезпечення безперебійної спільної роботи (режим Multiplayer) розробники застосували складну, але ефективну подієво-орієнтовану модель. Вона поєднує постійні WebSocket-з'єднання з брокером повідомлень Redis (патерн Pub/Sub) для миттєвої трансляції змін. Важливим архітектурним відкриттям стало застосування парадигми "Shared Code", яка дозволяє використовувати єдину кодову базу для клієнта та сервера, усуваючи дублювання бізнес-логіки та ідеально реалізуючи принцип DRY (Don't Repeat Yourself).
Також під час дослідження було оцінено переваги та обмеження обраних рішень. До головних переваг архітектури Penpot належить висока відмовостійкість, потокобезпечне управління станом завдяки функціональному програмуванню (незмінні структури даних) та використання відкритих вебстандартів (SVG як DOM-елементи). Серед потенційних обмежень можна виділити високий поріг входження для нових контриб'юторів через вузьку специфіку мови Clojure та загальну складність підтримки такої інфраструктури. Загалом, виконання цієї роботи дозволило сформувати міцні практичні навички архітектурного аналізу складного програмного забезпечення.

В.1 Довідка щодо генерації фрагмента коду управління станом (Atoms) 
Назва інструменту та версія: Gemini 3.1 Pro.
Мета використання: для створення ілюстративного псевдокоду, що демонструє роботу з потокобезпечними структурами даних (Atoms) у бекенді Penpot без класичного блокування потоків. 
Внесок автора: згенерований код адаптовано для звіту; додано цільові україномовні коментарі та видалено зайві імпорти бібліотек для фокусування виключно на механіці функції swap!.
Промпт: «Напиши простий фрагмент коду на Clojure, який показує, як безпечно оновити стан документа (додати фігуру) за допомогою atom та swap!. Зроби його коротким. »
Результат (приклад програмного коду):
  1 ;; Демонстрація потокобезпечного оновлення документа в пам'яті сервера
  2 (defonce current-document (atom {:shapes [] :selected nil}))
  3
  4 (defn add-shape-to-doc [new-shape]

В.2 Довідка щодо генерації фрагмента коду подієвої синхронізації (Pub/Sub) 
Назва інструменту та версія: Gemini 3.1 Pro.
Мета використання: для візуалізації архітектурного патерну Publish/Subscribe, який використовується в системі для забезпечення роботи Multiplayer-режиму через брокер Redis.
Внесок автора: спрощено структуру переданого об'єкта (payload); залишено лише базові параметри події для кращого академічного сприйняття принципу маршрутизації повідомлень.
Промпт: «Згенеруй короткий приклад коду на Clojure, що демонструє публікацію події (broadcast) в канал Redis для синхронізації макетів між користувачами.»
Результат (приклад програмного коду):
  1 ;; Демонстрація публікації оновлень макета
  2 (defn broadcast-update [project-id event-data]
  3   ;; Сервер публікує подію в канал конкретного проєкту через Redis
  4   (redis/publish! (str "project-channel:" project-id)
  5                   {:type :remote-update
  6                    :timestamp (current-time)
  7                    :payload event-data}))
