3 ОПИС ВИКОНАНОЇ РОБОТИ
У рамках виконання даної лабораторної роботи було здійснено доопрацювання документа «Vision & Scope» з акцентом на розділ 3 «Scope and Limitations» (див. додаток Б). Основною метою роботи було уточнення функціональних меж системи, деталізація архітектури початкового релізу та формування бачення подальшого розвитку із використанням технологій штучного інтелекту.
Об’єктом розробки виступає програмна система FoodPreOrder - SaaS-платформа для попереднього замовлення їжі та напоїв, яка дозволяє мінімізувати час очікування в чергах та оптимізувати процес обслуговування в закладах харчування.
У ході виконання роботи було реалізовано наступні завдання:
1. Деталізовано Scope of Initial Release (див. додаток Б), у межах якого описано повний функціонал системи та взаємодію її компонентів. Визначено архітектуру рішення, що включає серверну частину на C# (ASP.NET Core) із використанням MSSQL для збереження даних, а також Azure Blob Storage для роботи з медіаконтентом. Описано веб-панель адміністратора (React + TypeScript), мобільний додаток користувача (Flutter) та IoT-термінали кухні на базі Raspberry Pi 4. Сформовано цілісну модель функціонування системи як інтегрованої екосистеми.
2. Розроблено Scope of Subsequent Releases (див. додаток Б), де визначено напрямки розвитку системи з використанням штучного інтелекту. Проаналізовано можливості застосування ШІ для персоналізації рекомендацій, прогнозування навантаження на кухню та оптимізації обробки замовлень. Описано підходи до збору даних (історія замовлень, поведінкові патерни користувачів, часові характеристики) та визначено варіанти використання моделей - від інтеграції готових API до донавчання моделей на власних даних.
3. Сформовано Limitations and Exclusions (див. додаток Б), де визначено обмеження системи та обґрунтовано виключення частини функціоналу. Зокрема, враховано технічні обмеження IoT-пристроїв, складність впровадження повноцінних AI-рішень на початкових етапах, а також економічну доцільність реалізації окремих функцій.

3 SCOPE AND LIMITATIONS (РАМКИ ТА ОБМЕЖЕННЯ ПРОЄКТУ) 
3.1 Scope of Initial Release (Рамки первинного випуску)
Серверна частина виступає центральним елементом системи, побудованої на базі RESTful API. Сервер відповідає за цілісність даних, виконання бізнес-правил та управлінням процесів між усіма клієнтськими додатками. Розглянемо детальніше завдання сервера:
* архітектура даних та медіа: управління реляційною базою даних (MSSQL) для зберігання структурованої інформації: профілів користувачів, ієрархії меню, історії транзакцій; робота з хмарним сховищем (Blob Storage) для оптимізованого зберігання медіа-контенту (фотографій страв, логотипів), розвантажуючи основну базу від "важких" файлів; забезпечення мультимовності на рівні моделі даних: видача контенту (назви, описи) відповідно до мовного заголовка запиту (EN/UA).
* управління доступом та безпека: реалізація механізму автентифікації на основі JWT-токенів для захисту сесій; контроль ролей (Role-Based Access Control).
* управління каталогом (меню): забезпечення CRUD-операцій для формування меню: створення категорій, прив’язка інгредієнтів, налаштування модифікаторів (наприклад, вибір розміру порції); миттєве блокування можливості замовлення страв у всіх клієнтів при отриманні сигналу про відсутність інгредієнтів.
* обробка життєвого циклу замовлення: валідація вхідних даних кошика (перевірка актуальності цін, наявності позицій); інтеграція з платіжним шлюзом: ініціалізація транзакції та зміна статусу замовлення лише після підтвердження успішної оплати; автоматичний розподіл вхідних заявок на конкретні IoT-термінали відповідних закладів.
* API та документування: надання документованих точок доступу (Swagger UI) для взаємодії з мобільними та веб-клієнтами.
IoT-клієнт - програмне забезпечення для планшетів, OLED-дисплеїв або терміналів у зоні приготування, яке замінює принтери чеків та забезпечує двосторонній зв'язок "Кухня ⇔ Сервер". Детальніше про функціональні завдання IoT-клієнта: 
   1. відображення черги замовлень у форматі інтерактивних карток.
   2. деталізація кожної картки: склад замовлення, коментарі клієнта, номер чека, ім'я замовника.
   3. відображення динамічного таймера для кожного замовлення з кольоровою індикацією критичного часу очікування.
   4. зміна статусів в один дотик: переведення замовлення зі статусу "Нове" в "Готується", а потім у "Готово до видачі".
   5. синхронізація статусів: відправка сигналу на сервер для автоматичної генерації push-сповіщення клієнту (наприклад, "Ваша кава готова").
   6. звукові та візуальні сповіщення персоналу про надходження нових замовлень.
Веб-клієнт розробляється на бібліотеці React як швидка та зручна панель управління для власників бізнесу. Детальніше про функціональні завдання:
1. забезпечення єдиного центру для реєстрації підприємства, введення реквізитів та встановлення графіку роботи.
2. надання зрозумілого інтерфейсу для створення категорій, додавання нових страв та налаштування цін.
3. завантаження фотографій позицій меню з їхньою подальшою автоматичною оптимізацією системою для швидкого відображення у клієнтських додатках.
4. візуалізація бізнес-показників у вигляді наочних графіків, де відображається статистика продажів та визначаються пікові години навантаження.
5. забезпечення безпеки управління шляхом надання різних рівнів доступу до функціоналу (наприклад, повні права для власника та обмежені для найманого менеджера).
Мобільний додаток розробляється на платформі Flutter, що дозволяє створити швидкий, кросплатформний сервіс (iOS та Android), який є основним інструментом для гостя закладу. Детальніше про функціональні завдання мобільного клієнта:
1. забезпечення простого та швидкого процесу реєстрації або входу через номер телефону чи email.
2. використання платіжної системи для безпечної оплати замовлення в один дотик; збереження платіжних даних (картки) для прискорення майбутніх покупок.
3. підтримка постійного з'єднання з системою для моніторингу процесу приготування.
4. отримання та відображення миттєвих повідомлень (push-сповіщень) про актуальний статус замовлення (наприклад, "Готується", "Готово") прямо на екрані пристрою.

3.2 Scope of Subsequent Releases (Рамки наступних випусків)
У майбутніх релізах системи заплановано впровадження технологій штучного інтелекту (ШІ) для оптимізації бізнес-процесів закладів харчування та покращення клієнтського досвіду. Інтеграція ШІ охоплює три ключові аспекти: визначення сценаріїв застосування, організацію збору даних та вибір стратегії роботи з моделями.
3.2.1 Застосування (сценарії використання ШІ)
Штучний інтелект буде імплементовано для вирішення двох основних завдань, які приносять пряму вигоду як бізнесу, так і кінцевим споживачам. Першим напрямком є створення системи персоналізованих рекомендацій, спрямованої на підняття продаж. Безпосередньо у мобільному додатку алгоритми аналізуватимуть поточний склад кошика та попередню історію замовлень клієнта для формування релевантних пропозицій. Наприклад, при виборі певного напою система автоматично пропонуватиме найбільш підходящий десерт. Для закладу це дієвий інструмент збільшення середнього чека без додаткових витрат на маркетингові кампанії, тоді як клієнт отримує пришвидшений процес вибору страв та більш інтуїтивну навігацію по меню. Другим стратегічним напрямком стане прогнозування завантаженості закладу. Система глибоко аналізуватиме накопичені транзакції для точного передбачення кількості замовлень у конкретні години та дні тижня. Під час розрахунків алгоритм обов'язково враховуватиме додаткові змінні, такі як фактор сезонності, наближення свят та зміни погодних умов. Завдяки цій аналітиці адміністрація закладу зможе завчасно та обґрунтовано планувати графік виходу персоналу на зміну, а також точніше розраховувати обсяги закупівлі сировини. Це в комплексі дозволить суттєво мінімізувати псування залишків їжі та гарантувати виконання замовлень у заявлені терміни навіть у години пікового попиту.
3.2.2 Збирання даних (Data Collection Strategy)
Для забезпечення ефективної роботи моделей машинного навчання архітектура системи з все більшим масштабуванням системи підтримуватиме фоновий збір даних із різних своїх компонентів. На рівні сервера та бази даних відбуватиметься об’єднання транзакційної інформації: збереження точного складу кожного чека, загальної суми, вибраних модифікаторів порцій, часу проведення оплати та фактичного часу видачі готового замовлення. Паралельно мобільний клієнт акумулюватиме поведінкові дані, фіксуючи взаємодію користувача з інтерфейсом. Програма реєструватиме історію перегляду певних категорій меню, частоту пошукових запитів та відсоток відмов у випадках, коли товари були додані до кошика, але транзакція не завершилася успішно. Зі свого боку, кухонний IoT-клієнт надаватиме критично важливі операційні дані у вигляді еталонних міток часу. Оскільки персонал фізично фіксує перехід між статусами "Готується" та "Готово" на екрані термінала, система зможе з високою точністю вираховувати реальний час приготування кожної страви при різному рівні завантаженості виробництва. З метою безпеки та суворого дотримання політики конфіденційності персональна інформація, зокрема номери телефонів, імена та адреси електронної пошти, буде замінена на унікальні знеособлені ідентифікатори. Таким чином, штучний інтелект оперуватиме виключно сухими математичними шаблонами, не маючи жодного технічного доступу до приватного життя користувачів.
3.2.3 Навчання та використання (Model Strategy)
Для вирішення поставлених завдань застосовуватиметься гібридний підхід до роботи з моделями, що дозволить раціонально оптимізувати фінансові та часові витрати на розробку. У випадку системи персоналізованих рекомендацій розробка власної архітектури з нуля визнана економічно недоцільною. Замість цього система використовуватиме готові API хмарних ШІ-сервісів, наприклад, Azure Personalizer. Технічний процес передбачає, що сервер на базі ASP.NET Core відправлятиме через захищений канал знеособлені дані про поточну сесію, а хмарний сервіс миттєво повертатиме оптимізований список пропозицій. Якщо клієнт позитивно відреагує на рекомендацію та додасть страву до кошика, хмарна модель отримає сигнал підтвердження і автоматично скоригує свої внутрішні ваги для підвищення точності у майбутньому. Натомість для задачі прогнозування завантаженості кухні буде застосовано метод періодичного донавчання існуючих моделей. Оскільки операційні процеси та швидкість обслуговування є глибоко специфічними для кожного окремого закладу харчування, універсальні готові ШІ-сервіси демонструватимуть високу похибку. Для вирішення цієї проблеми використовуватимуться відкриті алгоритми аналізу часових рядів. Ці моделі будуть регулярно донавчатися безпосередньо на локальних транзакційних даних конкретного закладу, агрегованих з бази MSSQL. Після завершення циклу донавчання оптимізована модель розгортатиметься прямо у серверній частині системи. Це дозволить генерувати точні графіки прогнозів для відображення у веб-панелі адміністратора локально та без затримок, повністю виключаючи необхідність постійних звернень до сторонніх платних ресурсів.
3.3 Limitations and Exclusions (Обмеження та винятки)
Для збереження архітектурного фокусу та дотримання обраної бізнес-моделі, певний функціонал, який традиційно зустрічається у суміжних продуктах, був навмисно виключений з поточного та майбутніх релізів системи, а також визначено низку неочевидних технічних обмежень.
3.3.1 Відмова від готівкових розрахунків та післяплати (Exclusion)
Система не підтримуватиме можливість формування та відправки замовлення на кухню без його попередньої 100% онлайн-оплати через мобільний додаток. Хоча технічно реалізувати опцію "оплата готівкою на касі" нескладно, її виключено свідомо. По-перше, це повністю усуває фінансові ризики для закладів харчування (списання продуктів у випадках, коли клієнт зробив замовлення, але не прийшов). По-друге, обов'язкова цифрова оплата є критичною умовою для виконання головної бізнес-цілі проєкту - повної ліквідації черг біля каси, адже клієнт має просто забрати готовий пакунок, не витрачаючи час на транзакційні дії на місці.
3.3.2 Жорстка залежність від постійного інтернет-з'єднання (Limitation)
Зважаючи на архітектурний вибір на користь безперервної синхронізації статусів у реальному часі, повноцінна робота системи в офлайн-режимі не передбачена. Це неочевидне обмеження є критичним як для мобільного клієнта, так і для кухонного IoT-термінала. У разі втрати мережевого з'єднання IoT-екран на кухні не зможе отримувати нові замовлення, а клієнт - бачити оновлення статусів. Створення комплексних механізмів відкладеної (асинхронної) синхронізації для офлайн-роботи визнано недоцільним, оскільки це порушує саму ідею миттєвого гарантованого зв'язку між гостем та кухарем.
3.3.3 Обмеження ШІ щодо динамічного ціноутворення (Limitation)
Відповідно до планів розвитку (розділ 3.2), система збиратиме великі масиви даних та матиме модуль машинного навчання для високоточного прогнозування пікових годин попиту. Проте системі штучного інтелекту навмисно не буде надано програмного доступу до автоматичної зміни цін (surge pricing) у періоди високого навантаження. Хоча з технічної точки зору автоматичне підняття цін у години пік є логічним кроком для максимізації прибутку, у сфері громадського харчування такі алгоритми викликають різке падіння лояльності споживачів. Таким чином, управління ціновою політикою строго ізольоване від алгоритмів ШІ і залишається за менеджментом закладу через веб-панель адміністратора.
