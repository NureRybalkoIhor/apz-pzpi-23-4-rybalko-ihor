	2 ЗАВДАННЯ
На цій лабораторній роботі потрібно розробити front-end частину програмної системи. Ця частина системи повинна підтримувати роботу клієнтів у веб-режимі та забезпечувати адміністрування системи, тобто мати окремий інтерфейс для звичайних користувачів і окремий - для адміністраторів (відповідно до ролей користувачів вашої системи).
Користувацький інтерфейс та інтерфейс адміністрування повинні підтримувати інтернаціоналізацію та локалізацію, а саме, забезпечувати: підтримку української на англійської мов сайту, формат дати та часу відповідний регіону, порядок сортування текстових значень, різні напрями введення тексту тощо. 
Звіт повинен містити опис прийнятих інженерних рішень, будову застосунку та опис його взаємодії з іншими частинами системи. Обов'язковими є UML діаграма прецедентів та UML діаграма компонент. Окрім цього, необхідно додати не менше двох діаграм на вибір із наступних: ER-модель даних мобільної платформи, UML діаграма пакетів, UML діаграма взаємодії, UML діаграма діяльності або UML діаграма станів. Код застосунку необхідно завантажити на GitHub до захисту роботи.
3 ОПИС ВИКОНАНОЇ РОБОТИ
Під час виконання лабораторної роботи було вирішено завдання проєктування та реалізації веб-клієнтського застосунку для платформи попереднього замовлення їжі FoodPreOrder. Розробка зосередилася на побудові повнофункціонального SPA (Single Page Application), який забезпечує інтерфейси для трьох категорій користувачів: клієнтів закладів, власників ресторанів із кухонним персоналом та системних адміністраторів. Розробка велася мовою TypeScript із використанням бібліотеки React 19, збирача Vite 8, бібліотеки управління станом Zustand та фреймворку інтернаціоналізації i18next, що дозволило забезпечити реактивний інтерфейс із підтримкою двох мов та темної/світлої теми.
Реалізація веб-застосунку охопила сценарії взаємодії всіх ролей із системою: від реєстрації та авторизації до адміністрування користувачів, управління ресторанами та меню, оформлення замовлень клієнтами та відстеження їх статусу кухонним персоналом. Також було реалізовано підсистему аналітики з візуалізацією продажів та піковим навантаженням через інтерактивні графіки.
3.1 Архітектура веб-застосунку
Архітектура застосунку побудована за компонентним принципом із розмежуванням відповідальності, що структурно проєкт складається з чотирьох логічних шарів.
Шар презентації (Presentation Layer) реалізований за допомогою React 19 та містить десять сторінок: Home, Login, Register, Restaurants, RestaurantMenu, Cart, Orders, Profile, AdminDashboard та BusinessDashboard. Навігація між сторінками здійснюється через React Router v7 з параметризованими маршрутами. Компонент MainLayout забезпечує єдину точку рендерингу навігаційної панелі, яка адаптується залежно від ролі користувача: публічна (для гостей та клієнтів), адміністративна (червоний стиль) та бізнесова (зелений стиль). Компонент ProtectedRoute реалізує захист маршрутів за ролями - неавторизований користувач перенаправляється на сторінку входу, а авторизований із невідповідною роллю - на головну.
Шар управління станом (State Layer) побудований на бібліотеці Zustand із middleware persist для збереження даних між сесіями. Реалізовано три стори: authStore (JWT-токен, дані користувача, стан авторизації), cartStore (позиції кошика, прив'язка до ресторану, підрахунок суми та кількості) та settingsStore (мова інтерфейсу, тема оформлення). Стор authStore зберігає мінімальний набір даних користувача (email, fullName, role), отриманих із відповіді сервера при авторизації. Стор cartStore реалізує бізнес-правило монозакладу - при додаванні страви з іншого ресторану кошик автоматично очищується.
Шар сервісів (Service Layer) побудований на бібліотеці axios із централізованою конфігурацією. Модуль api.ts ініціалізує HTTP-клієнт із базовою URL-адресою серверу (http://localhost:5082/api) та реєструє два interceptors: request-interceptor, який автоматично додає заголовок Authorization із Bearer JWT-токеном до кожного запиту, та response-interceptor, який при отриманні HTTP 401 виконує автоматичний logout і редирект на сторінку входу. Реалізовано сім спеціалізованих сервісів: AuthService (реєстрація, авторизація, профіль), AdminService (управління користувачами, блокування, зміна ролей, логи), RestaurantService (CRUD ресторанів), CategoryService (CRUD категорій з фільтрацією по ресторану), DishService (CRUD страв із підтримкою multipart/form-data для завантаження зображень), OrderService (створення, оплата, зміна статусу замовлень) та StatsService (статистика продажів).
Шар інтернаціоналізації (i18n Layer) побудований на бібліотеці i18next із плагіном react-i18next. Підтримуються дві мови: українська (uk) та англійська (en), кожна визначена у окремому JSON-файлі з приблизно 160 ключами перекладу. Мовні файли охоплюють усі елементи інтерфейсу: назви сторінок, лейбли форм, статуси замовлень, назви ролей, повідомлення про помилки та порожні стани. Для форматування дат використовується бібліотека date-fns із відповідною локаллю (uk або enUS). Двомовність поширюється також на контент: усі сутності (ресторани, категорії, страви) мають поля nameUA/nameEN та descriptionUA/descriptionEN, які відображаються відповідно до обраної мови.
3.2 Реалізація клієнтського інтерфейсу для ролі Customer
Роль Customer є основною цільовою аудиторією системи і охоплює повний цикл попереднього замовлення їжі.
Після реєстрації (fullName, email, phone, password із валідацією підтвердження пароля) та авторизації клієнт перенаправляється на каталог ресторанів. Сторінка Restaurants завантажує список закладів через GET /api/restaurants та відображає їх у адаптивній сітці карток із зображенням, назвою (відповідно до мови) та адресою. Реалізовано текстовий пошук за назвою та адресою ресторану.
Сторінка RestaurantMenu відображає меню обраного ресторану із системою вкладок по категоріях. Завантаження виконується паралельно через Promise.all (дані ресторану, категорії, страви). Кожна страва відображається як картка з фото, назвою, описом, ціною та кнопкою додавання в кошик. Для неавторизованих користувачів замість кнопки «Додати» відображається кнопка «Увійти».
Сторінка Cart реалізує повнофункціональний кошик із контролем кількості (кнопки «+»/«−»), видаленням окремих позицій, вибором часу візиту (input type=time з автоматичним перенесенням на наступний день при виборі минулого часу), опціональним коментарем та підрахунком суми. При оформленні замовлення формується POST-запит до /api/orders із restaurantId, visitTime (ISO 8601), масивом позицій (dishId, quantity) та коментарем.
Сторінка Orders відображає історію замовлень із кольоровими бейджами статусів (Pending - жовтий, Paid - синій, Cooking - фіолетовий, Ready - зелений, Completed - зелений, Cancelled - червоний). Для замовлень зі статусом Pending доступна кнопка оплати, яка відправляє POST /api/orders/{id}/pay. Дати форматуються з урахуванням обраної локалі.
Сторінка Profile відображає аватар (перша літера імені), повне ім'я, email та роль із перекладом.
3.3 Реалізація бізнес-панелі для ролей RestaurantOwner та KitchenStaff
Бізнес-панель доступна за маршрутом /business і забезпечує управління рестораном через інтерфейс із бічною панеллю навігації (sidebar) та чотирма вкладками.
Вкладка «Інформація про ресторан» відображає дані обраного ресторану: назви UA/EN, адресу, координати, статус (active/blocked), дані власника та зображення. При наявності кількох ресторанів у сайдбарі відображається селектор для перемикання. Роль RestaurantOwner має можливість створити новий ресторан через форму (назва UA/EN, адреса, координати).
Вкладка «Управління меню» забезпечує повний CRUD-цикл для категорій та страв. Категорії створюються з двомовними назвами (UA/EN) та прив'язкою до ресторану через POST /api/categories. Видалення категорії каскадно видаляє пов'язані страви з інтерфейсу. Додавання страви реалізовано через модальне вікно з полями: назва UA/EN, опис UA/EN, ціна, час приготування, категорія (select), зображення (file input). Форма відправляється як FormData (multipart/form-data) через POST /api/dishes, що забезпечує завантаження зображень. Реалізовано валідацію з підсвічуванням полів (назва та категорія обов'язкові) та toast-повідомленнями. Кожна страва відображається із toggle-бейджем доступності (Available/Unavailable).
Вкладка «Замовлення» відображає таблицю всіх замовлень обраного ресторану з колонками: ID, позиції, сума, час візиту, дата створення, статус та дії. Управління статусами реалізовано через кнопки, які відправляють PATCH /api/orders/{id}/status відповідно до бізнес-потоку: Paid→Cooking→Ready→Completed. Додатково доступна кнопка скасування (Cancelled) для замовлень у статусах Pending, Paid та Cooking.
Вкладка «Аналітика» побудована на реальних даних замовлень та відображає: чотири stat-картки (загальні продажі, кількість замовлень, середній чек, кількість страв), лінійний графік продажів по годинах (LineChart, Recharts), стовпчикову діаграму кількості замовлень по годинах (BarChart) та кругову діаграму розподілу продажів по категоріях (PieChart). Дані для графіків обчислюються на клієнті шляхом агрегації замовлень.
Розмежування прав між RestaurantOwner та KitchenStaff реалізовано на рівні UI: кнопки створення ресторану, видалення категорій та видалення страв відображаються лише для RestaurantOwner (перевірка user.role === 'RestaurantOwner'). KitchenStaff має доступ на перегляд усієї інформації та управління статусами замовлень.
3.4 Реалізація адміністративної панелі для ролі Admin
Адміністративна панель доступна за маршрутом /admin і забезпечує системне управління через інтерфейс із бічною панеллю та трьома вкладками.
Вкладка «Огляд системи» відображає три stat-картки (загальна кількість користувачів, активних, заблокованих) та таблицю перших п'яти користувачів із аватаром, email, роллю та статусом.
Вкладка «Управління користувачами» завантажує повний список через GET /api/admin/users і відображає таблицю з пошуком (за email або ім'ям) та фільтрацією за роллю (Admin, RestaurantOwner, KitchenStaff, Customer). Для кожного користувача доступні дві дії: блокування/розблокування (POST /api/admin/block та POST /api/admin/unblock/{userId}) та зміна ролі через inline-select (PUT /api/admin/users/role). Зміна ролі активується кліком на бейдж поточної ролі.
Вкладка «Управління даними» реалізує функції резервного копіювання: експорт усіх користувачів у JSON-файл (із метаданими exportedAt та version) та імпорт із раніше збереженого JSON. Експорт виконується через створення Blob та програмний клік на посилання, імпорт - через FileReader із парсингом JSON та валідацією структури.
3.5 Підсистема інтернаціоналізації та тематизації
Підтримка двох мов (українська та англійська) реалізована на двох рівнях. Статичний контент інтерфейсу (лейбли, кнопки, повідомлення, назви ролей та статусів) перекладається через функцію t() із i18next. Динамічний контент (назви ресторанів, категорій, страв, описи) відображається через утилітні функції getName() та getDesc(), які обирають відповідне поле (nameUA/nameEN) залежно від поточної мови.
Перемикання мови здійснюється кнопкою у навігаційній панелі та зберігається у settingsStore (localStorage). Дати форматуються через date-fns із автоматичним підбором локалі.
Підтримка темної та світлої теми реалізована через CSS-змінні на кореневому елементі (data-theme attribute). При перемиканні теми функція applyTheme() миттєво оновлює атрибут без перезавантаження сторінки. Обрана тема зберігається у localStorage і відновлюється при наступному відвідуванні.
3.6 Використання засобів ШІ
3.6.1 Довідка щодо генерації системи HTTP-інтерсепторів із автоматичною авторизацією та обробкою помилок за допомогою ШІ
Назва інструменту та версія: Claude Sonnet 4.6 (Thinking).
Мета використання: для реалізації централізованої системи axios-інтерсепторів, яка автоматично додає JWT Bearer-токен до кожного HTTP-запиту та виконує автоматичний logout із редиректом при отриманні відповіді з кодом 401 (Unauthorized), що забезпечує єдину точку управління авторизацією без дублювання логіки в окремих компонентах.
Внесок автора: згенерований код було адаптовано до архітектури проєкту - інтегровано взаємодію із Zustand-стором authStore для отримання токена через getState() (без підписки на реактивні оновлення) у request-інтерсепторі та виклику методу logout() у response-інтерсепторі. Додано перевірку наявності заголовків у конфігурації запиту перед встановленням Authorization. Розширено обробку помилок із програмним редиректом через window.location.href для гарантованого переходу на сторінку входу незалежно від стану React Router. Реалізовано сім спеціалізованих API-сервісів (AuthService, AdminService, RestaurantService, CategoryService, DishService, OrderService, StatsService) із типізацією відповідей через TypeScript generics.
Промпт: «Дій як React TypeScript Developer. Напиши модуль api.ts на базі axios, який створює HTTP-клієнт із двома інтерсепторами: request-інтерсептор додає Bearer JWT-токен із Zustand-стору, response-інтерсептор при 401 викликає logout() і редиректить на /login. Базовий URL: http://localhost:5082/api.»
Приклад коду дивитись у додатку В.1
3.6.2 Довідка щодо генерації стору кошика з бізнес-правилом монозакладу за допомогою ШІ
Назва інструменту та версія: Claude Sonnet 4.6 (Thinking).
Мета використання: для реалізації реактивного стору кошика на базі Zustand із middleware persist, який інкапсулює бізнес-правило монозакладу - автоматичне очищення кошика при додаванні страви з іншого ресторану - та забезпечує персистентне зберігання стану між сесіями через localStorage.
Внесок автора: згенерований код було адаптовано під типи даних серверної частини (числові ідентифікатори замість рядкових). Розширено логіку методу removeItem - при видаленні останньої позиції кошик автоматично скидає прив'язку до ресторану (restaurantId та restaurantName встановлюються у null). Додано обчислювані методи totalPrice() та totalItems() через патерн get() для доступу до актуального стану всередині persist-middleware. 4 ВИСНОВКИ
Під час виконання лабораторної роботи було проведено повний цикл проєктування та програмної реалізації веб-клієнтського застосунку для платформи попереднього замовлення їжі FoodPreOrder. Основною метою роботи було створення повнофункціонального SPA, який забезпечує рольовий доступ для трьох категорій користувачів - клієнтів закладів, власників ресторанів із кухонним персоналом та системних адміністраторів - та безперебійну взаємодію із серверною частиною через RESTful API. У процесі розробки було застосовано сучасні підходи до побудови веб-застосунків, включаючи бібліотеку React 19 з TypeScript, збирач Vite 8, бібліотеку реактивного управління станом Zustand та фреймворк інтернаціоналізації i18next із підтримкою двох мов.
Ключовим етапом роботи стала реалізація рольового розмежування інтерфейсів із чіткою спеціалізацією кожного. Для ролі Customer було реалізовано повний цикл попереднього замовлення: каталог ресторанів із пошуком, меню із фільтрацією по категоріях, кошик із контролем кількості та вибором часу візиту, оформлення замовлення та оплата. Для ролей RestaurantOwner та KitchenStaff було побудовано бізнес-панель із чотирма функціональними модулями: інформація про ресторан, CRUD-управління меню (категорії та страви з двомовними назвами та завантаженням зображень через FormData), управління статусами замовлень та аналітичний дашборд із інтерактивними графіками на основі реальних даних (LineChart, BarChart, PieChart бібліотеки Recharts). Для ролі Admin реалізовано панель системного адміністрування з пошуком та фільтрацією користувачів, блокуванням/розблокуванням, зміною ролей та резервним копіюванням даних у JSON.
Важливим архітектурним рішенням стала побудова шару сервісів на базі axios із централізованими interceptors: request-interceptor автоматично додає JWT-токен до кожного запиту, а response-interceptor обробляє HTTP 401, виконуючи автоматичний logout та редирект. Такий підхід забезпечує єдину точку управління авторизацією без дублювання логіки у компонентах. Захист маршрутів реалізовано через компонент ProtectedRoute, який перевіряє наявність авторизації та відповідність ролі, забезпечуючи неможливість доступу до адміністративних або бізнес-сторінок без належних повноважень.
Підсистема інтернаціоналізації охоплює як статичний контент інтерфейсу (близько 160 ключів перекладу), так і динамічний контент бази даних (двомовні назви та описи сутностей). Форматування дат виконується через бібліотеку date-fns із автоматичним підбором локалі. Підтримка темної та світлої теми реалізована через CSS-змінні із миттєвим перемиканням без перезавантаження сторінки.
Отримана програмна реалізація повністю відповідає функціональним вимогам, забезпечує стабільну взаємодію із серверною частиною системи (ASP.NET Core API на порті 5082) та готова до демонстрації на захисті.
